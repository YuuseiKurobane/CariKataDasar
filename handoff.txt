2. User's goal and definition of failure
----------------------------------------

The user is developing Yomitan's Bahasa Indonesia parser/candidate generator.
The central concern is not that the parser generates implausible or fake
candidates. The important failures are:

1. A real Indonesian headword is absent from the generated candidates.
2. A real candidate exists but appears in a bad position.

The main empirical debugging source should be real Indonesian text from the
wild:

1. Start with the mc4-id token frequency table.
2. Discard tokens below a configurable occurrence threshold (default 100).
3. Discard tokens already present exactly in the combined headword set.
4. Run each remaining token through the Yomitan-compatible Indonesian
   candidate generator.
5. Find the first generated candidate that is a known headword.
6. Produce a report exposing the complete unique candidate order.

Community-reported difficult words are the second major regression source.


3. Original requested report
----------------------------

The user's original proposed shape was:

    token,resolution,num,word1,word2,word3,...

Where:

- token is the raw corpus token.
- resolution is the first generated candidate found in the headword set.
- num is the position of that candidate.
- word1, word2, ... are candidates in Yomitan lookup order.

The agreed/recommended expanded human-facing CSV is:

    token,occurrences,resolution,resolution_rank,resolution_sources,candidate_count,word1,word2,...

Important semantics:

- word1 is the original raw token.
- Rankings are over first-occurrence UNIQUE candidate strings.
- Candidate order must never be alphabetically sorted.
- resolution and resolution_rank should be blank when no candidate resolves.
- occurrences must be retained so debugging can be prioritized.
- resolution_sources records which input dictionaries contain the result.

A compressed long-form JSONL candidate trace is also recommended as the
machine-readable source of truth, since a dynamic wide CSV is awkward to
stream and loses transform/preprocessor details.

This matters because an experiment runner can stop reading as soon as

7. Current implementation
-------------------------

src/candidate-generator.js

- Creates a `LanguageTransformer` with the Indonesian descriptor.
- Reproduces Indonesian's configured Yomitan preprocessing:
  - decapitalize
  - capitalize first letter
  - remove alphabetic diacritics
- Preserves Yomitan text-variant ordering.
- Runs each variant through the transformer.
- Deduplicates candidate text at first occurrence.
- Returns rank, text, preprocessing chain, transform chain, and conditions.

Candidate generation is breadth-first inside Yomitan's LanguageTransformer.
Transform group order and rule order are therefore behavior and must not be
changed casually.

The transformer can generate empty strings and linguistically silly strings.
This is currently accepted because the user's explicit priority is coverage
and ordering of real candidates, not suppression of fake candidates.

The harness intentionally processes one already-tokenized corpus token. It
does not reproduce Yomitan's arbitrary substring/prefix scanning of a larger
text selection. That omission is intentional for this experiment.


src/headword-index.js

- Reads every `.txt` file in `data/headwords/sources`, sorted by filename.
- Builds an exact `Set` of headwords.
- Builds a map from headword to every source filename containing it.
- Preserves source provenance rather than collapsing all dictionaries into an
  opaque union.


src/report-generator.js

- Resolves ONE token against a loaded headword index.
- Selects the first matching unique candidate.
- Produces an object with resolution rank and source dictionaries.
- Can flatten that object into the proposed wide-report record.

This is not yet a frequency-file runner.


scripts/build-headwords.mjs

- Builds the two generated headword artifacts atomically.
- Already ran successfully and wrote 276,540 exact headwords.


scripts/check-upstream.mjs

- Verifies the pinned, non-editable Yomitan compatibility files by SHA-256.
- Does not verify `indonesian-transforms.js` because that is intentionally the
  main editable parser file.

12. Recommended implementation for the experiment CLI
------------------------------------------------------

Add:

    scripts/run-experiment.mjs

Add to package.json:

    "experiment": "node scripts/run-experiment.mjs"

Recommended CLI options:

    --threshold-minimum 100
    --frequencies tools/corpus/mc4-id/run-v1/token_frequencies.csv.gz
    --headwords data/headwords/sources
    --output reports/debug.csv.gz
    --trace-output reports/candidate-trace.jsonl.gz

The default threshold should be 100 to match the user's original prompt.

Required processing:

1. Load the exact headword index once.
2. Stream-decompress the frequency CSV; do not load 38.9 million rows into
   memory.
3. Validate the exact header `token,occurrences`.
4. Because rows are frequency-sorted, stop when occurrences < threshold.
5. Skip the token if it is already an exact headword.
6. Generate unique ordered candidates using `IndonesianCandidateGenerator`.
7. Resolve to the first candidate present in the exact headword set.
8. Preserve the inherited frequency order in output.
9. Emit run statistics:
   - eligible token count
   - exact-headword skips
   - parser-resolved count
   - parser-unresolved count
   - occurrence-weighted equivalents
   - resolution-rank histogram
10. Write outputs atomically.

CSV parsing must be real CSV parsing or carefully validated. Do not assume
that splitting every line on commas is universally safe merely because the
current tokenizer usually excludes commas.


13. Wide CSV streaming decision
-------------------------------

The wide CSV has a dynamic number of `wordN` columns. A streaming writer does
not know the maximum candidate count when writing the header.

Reasonable solutions:

A. Recommended:
   - Stream each complete result to temporary JSONL.
   - Track the maximum candidate count.
   - Perform a second pass to write the final wide CSV header and rows.
   - Optionally keep a compressed detailed JSONL trace.

B. Accept a `--max-candidates` report width and put full traces in JSONL.

C. Use only normalized long-form CSV:

       token,occurrences,candidate_rank,candidate,is_headword,sources,...

   This is easier to stream but does not exactly match the user's requested
   spreadsheet.

Do not buffer all 578,204 result objects in RAM merely to discover the widest
row.

15. Later improvements, not prerequisites for the first run
------------------------------------------------------------

Context extraction:

The frequency table contains token and count only. It discarded surrounding
sentences. A high-frequency unresolved token may be Indonesian, a proper name,
another language, boilerplate, or web debris.

After the first report exists, add a second-pass context extractor for a
selected set of high-priority unresolved tokens. It should rescan mc4-id and
retain a few representative sentences (and URL if available) per target.
Do not rerun or redesign the frequency count merely to get context.

Diff reports:

For parser iteration, compare old and new runs:

- newly resolved real tokens
- newly missing tokens
- resolution rank improved/worsened
- occurrence-weighted coverage change

This is more useful than looking only at total candidate count.

Community ordering:

Extend community JSONL with explicit fields such as `expected_first`,
`must_precede`, or a maximum acceptable rank only when the desired ordering is
known. Keep simple coverage assertions separate from ordering assertions.
